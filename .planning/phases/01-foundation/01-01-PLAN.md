  <action>
Write tests first (RED), then implement TimerEngine (GREEN).

**TimerEngine.swift** -- implement exactly as specified in RESEARCH.md Pattern 1:
- `@MainActor final class TimerEngine: ObservableObject`
- States: `idle, running, paused, onBreak`
- Modes: `work, break_`
- `@AppStorage("workDuration") var workDuration: Int = 25`
- `@AppStorage("breakDuration") var breakDuration: Int = 5`
- `@Published var timerState`, `@Published var timeRemaining`, `@Published var currentMode`
- Private: `startDate: Date?`, `targetDuration: Int`, `ticker: Timer?`
- Methods: `start()`, `pause()`, `stop()`, `tick()`, `handleSessionComplete()`, `onWake()`
- Clock-based elapsed time: `timeRemaining = max(0, targetDuration - Int(Date().timeIntervalSince(startDate)))`
- Sleep/wake: subscribe to `NSWorkspace.didWakeNotification` in start(), call tick() on wake
- Auto-transition: when timeRemaining == 0 in work mode, switch to break_ and auto-start break timer
- When break completes: return to idle, set currentMode = .work, reset timeRemaining to workDuration * 60

**Testing note:** For clock-based tests, inject a small targetDuration (e.g., 2 seconds) and use XCTestExpectation with a short timeout. For state transition tests, call methods directly and assert state changes.

Do NOT use `timeRemaining -= 1` anywhere. The clock is the source of truth.
